# 用户权利响应（数据主体权利）

> 范围：数据主体（患者 / 用户）对本系统中其 PHI 所享有的权利及本系统的响应能力。事实锚点：`safety.patient-profile.ts`、`lib/phi-crypto.mjs`、`audit-chain.mjs`。

## 权利清单与落地状态

| 权利 | 状态 | 实现 / 响应机制 |
|---|---|---|
| **被遗忘权（Erasure）** | ✅ 已落地 | `forget_patient` 工具 → `secureWipeFile` 覆写+删除密文；须 `confirm=true` 防误触；审计仅记动作不记 PHI（E2 闭环） |
| **访问权（Access）** | ⚠️ 部分 | 患者画像写入经 `remember_patient`，字段名记于审计日志（`patient_profile.write` + `fields`）；**无独立「导出我的数据」工具**，需经运维从密文解密获取 |
| **更正权（Rectification）** | ✅ 经既有机制 | `remember_patient` 字符串字段覆盖更新、数组字段增量合并去重，用户可随时重新陈述以更正 |
| **限制/反对处理** | ⚠️ 设计内 | 系统不主动外传 PHI 训练；用户可经 `forget_patient` 终止本地留存以实质限制处理 |
| **可携带权（Portability）** | ❌ 未实现 | 无结构化导出工具；密文格式非通用，需解密后人工交付 |

## 被遗忘权响应流程（标准作业）

1. 用户明确要求删除其本地患者画像。
2. Agent 调用 `forget_patient` 并显式 `confirm=true`（防误触）。
3. 系统经 `secureWipeFile` 对 `.pi/patient-profile.json` 覆写随机字节 3 次后删除。
4. 审计记录 `patient_profile.forget`（动作 + `wiped:true`），**不含任何 PHI 原值**。
5. 后续对话 `on("context")` 加载空画像，不再注入该用户画像。

> 响应时效：同步即时（工具调用即生效），符合「无不合理延迟」原则。

## 已知缺口（透明披露）

- **无自助访问/导出工具**：用户若要求「导出我的全部数据」，当前需经运维从 `.pi/patient-profile.json` 解密 + 审计日志聚合，无自助端点。建议后续补 `export_patient` 工具（同样须确认门 + 审计）。
- **无书面权利申请受理台账**：权利请求目前依赖对话中的显式指令，未建独立受理/追踪系统。生产化前建议补工单式受理。

## 设计哲学

本系统以「**本地优先 + 最小字段 + 加密留存 + 可被遗忘**」为隐私基线：患者 PHI 默认仅存于本地加密文件，绝不入库、绝不训练外传；一旦用户行使被遗忘权，密文经安全擦除彻底不可恢复。此设计使多数权利可经「删除即终止处理」自然满足，而非依赖复杂的跨系统编排。
